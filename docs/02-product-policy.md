# 자린고비 서비스 운영 정책

- 문서 상태: Draft v0.2
- 작성일: 2026-07-10
- 대상: MVP 제품 동작과 운영 기준
- 반영 기준: 2026-07-10 사용자 확정사항
- 주의: 제품 정책 초안이며 이용약관·개인정보 처리방침을 대신하지 않는다.

## 1. 정책 목표

- 공휴일과 중도 합류에도 모든 참여자에게 같은 계산식을 적용한다.
- 방 생성 후 계산 조건이 바뀌지 않도록 기간과 기준금액을 고정한다.
- 종료, 보정, 정산, 완료의 시간 경계를 겹치지 않게 정의한다.
- 지출 변경을 잠근 뒤 결과와 이전 기록을 장기간 일관되게 열람하게 한다.
- 사진과 댓글을 공유하되 개인 지출 기록과 방 공개 정보의 경계를 명확히 한다.

## 2. 확정 기본 정책

| 항목 | MVP 정책 |
|---|---|
| 통화·시간대 | KRW, Asia/Seoul |
| 기간 | 1~31일의 선택일, 시작일·종료일 포함 |
| 기준금액 | 방 생성 시 입력하고 즉시 고정 |
| 공휴일 | 방 생성 시점의 대한민국 공휴일·대체공휴일을 제외 |
| 적용한도 | 기준금액에 공휴일과 합류일을 비례 반영 |
| 정원 | 생성 시 설정하고 방장이 서비스 최대값 안에서 증가 가능 |
| 참여 | 예정 및 진행 중 참여 가능, 종료 경계부터 불가 |
| 수입 | 입력 기능 제공하지 않음 |
| 카테고리 | 점심, 커피, 간식, 저녁, 필수품, 사치품 |
| 사진 | 챌린지 지출당 정확히 1장 필수 |
| 보정 | 종료 후 12시간까지 기간 내 지출 입력·수정·삭제 가능 |
| 정산 | 보정 종료 후 36시간 동안 지출 변경 잠금 |
| 완료 기록 | 자동 만료 없이 읽기 전용 보관 |
| 왕관 | 남은 금액이 가장 큰 활성 멤버에게 표시, 동률은 공동 표시 |

## 3. 계정과 프로필

### 3.1 계정

- 기기 변경과 기록 보관을 위해 지속 가능한 계정이 필요하다.
- 한 계정은 같은 방에 한 번만 참여할 수 있다.
- 인증 수단은 기술·스토어 정책 단계에서 확정한다.

### 3.2 프로필

- 방에는 닉네임과 프로필 이미지를 공개한다.
- 닉네임은 2~20자이며 공백만으로 구성할 수 없다.
- 왕관은 닉네임 문자열 자체를 바꾸지 않고 방 화면에서만 앞에 붙이는 표현 속성이다.
- 욕설, 사칭, 연락처 노출 등 운영 기준을 위반하는 프로필은 변경 요청 또는 제한할 수 있다.

## 4. 챌린지 시간과 생명주기

### 4.1 시간 기호

방 시간대는 MVP에서 `Asia/Seoul`로 고정한다.

- `S`: 시작일 00:00
- `E`: 화면에서 선택한 종료일 다음 날 00:00
- `C = E + 12시간`: 지출 보정 마감 및 정산 잠금 시작
- `F = E + 48시간`: 정산 종료 및 완료 시각

종료일은 사용자에게 포함 날짜로 보이지만 시스템 계산 구간은 `[S, E)`다.

### 4.2 상태와 권한

| 상태 | 시간 구간 | 지출 | 댓글·답글 | 참여 |
|---|---|---|---|---|
| 대기 | `t < S` | 챌린지 지출 불가 | 기존 스레드 없음 | 가능 |
| 진행 | `S ≤ t < E` | 입력·수정·삭제 | 작성·5분 내 수정·삭제 | 가능 |
| 보정 | `E ≤ t < C` | 기간 내 지출 입력·수정·삭제·사진 재업로드 | 작성·5분 내 수정·삭제 | 불가 |
| 정산 | `C ≤ t < F` | 읽기 전용 | 작성·5분 내 수정·삭제 | 불가 |
| 완료/보관 | `F ≤ t` | 읽기 전용 | 읽기 전용, 본문 삭제 요청 가능 | 불가 |

예: 월요일~금요일 챌린지는 토요일 00:00에 보정으로 바뀐다. 토요일 12:00에 지출을 잠그고 일요일 24:00까지 정산한 뒤 월요일 00:00에 완료한다.

### 4.3 생성과 불변 조건

방 생성 성공 시 다음 값을 하나의 계산 조건 스냅샷으로 고정한다.

- 시작일과 종료일
- 선택일 목록과 날짜 선택 방식
- 기준금액
- 통화와 방 시간대
- 공휴일 데이터 버전과 제외 공휴일 목록

위 값은 방장도 대기·진행·보정·정산·완료 상태에서 수정할 수 없다. 방 이름, 프로필, 정원, 참여코드 같은 비계산 항목만 별도 정책에 따라 바꿀 수 있다.

입력 실수는 시작 전이고 공유 지출이 없는 상태에서 방장이 방을 삭제한 뒤 새 방을 만드는 방식으로 처리한다. 방을 만들기 전 최종 확인 화면에서 기간, 기준금액, 제외 공휴일, 예상 적용한도를 다시 보여준다.

### 4.4 방 삭제와 보관 보호

- 시작 전이고 공유 지출이 없을 때만 방장이 방을 삭제할 수 있다.
- 시작 후에는 다른 참여자의 기록 보존을 위해 방장이 전체 방을 일괄 삭제할 수 없다.
- 완료 방은 각 사용자가 자신의 목록에서 숨길 수 있으나 다른 사용자의 기록은 삭제되지 않는다.
- 개인정보 삭제 요청은 해당 사용자의 사진·메모·댓글 본문을 제거하고 `삭제된 콘텐츠` 표식과 비식별 결과 스냅샷을 남기는 방식으로 처리한다.

## 5. 공휴일과 적용한도 계산

### 5.1 날짜 집합

- `A`: 방장이 선택한 전체 날짜 집합
- `N = |A|`: 공휴일 제외 전 전체 선택일 수
- `H`: 방 생성 시점에 확인된 대한민국 공휴일·대체공휴일 집합
- `D = A - H`: 실제 유효 챌린지 날짜 집합
- `J_i`: 참여자 i의 서버 기준 참여 확정 시각
- `R_i`: `D` 중 참여자 i의 합류일 이상인 날짜 수

`다음 평일` 프리셋은 월요일~금요일 5개 날짜를 `A`로 만든다. 직접 선택한 기간은 주말도 선택일에 포함하며, 공휴일만 자동 제외한다.

선택일이 모두 공휴일이라 `D`가 비면 방을 생성할 수 없다.

### 5.2 공휴일 스냅샷

- 방 생성 시점에 알려진 대한민국 공휴일, 대체공휴일, 임시공휴일을 저장한다.
- 생성 후 새 임시공휴일이 지정돼도 기존 방의 날짜와 적용한도는 소급 변경하지 않는다.
- 제외 공휴일은 생성 확인, 참여 확인, 방 정보, 지난 기록에서 계속 표시한다.
- 공휴일에 발생한 지출은 개인 지출 기록에는 남길 수 있지만 해당 챌린지에는 연결할 수 없다.

### 5.3 개인별 적용한도

- `B`: 방의 기준금액
- `L_i = floor(B × R_i ÷ N)`: 참여자 i의 적용한도
- `X_i`: 참여자 i의 유효 챌린지 지출 합계
- `M_i = L_i - X_i`: 참여자 i의 남은 금액
- `P_i = X_i ÷ L_i × 100`: 참여자 i의 진행률

원 단위 미만은 버린다. 계산은 부동소수점 오차가 없도록 정수 연산으로 처리한다. `L_i`가 0원이면 0원 챌린지 참여로 명확히 안내하며 지출 1원부터 초과로 표시한다.

| 조건 | 계산 | 적용한도 |
|---|---:|---:|
| 5일·50,000원, 공휴일 없음, 시작 전 참여 | 50,000 × 5 ÷ 5 | 50,000원 |
| 5일·50,000원, 공휴일 1일, 시작 전 참여 | 50,000 × 4 ÷ 5 | 40,000원 |
| 5일·50,000원, 공휴일 없음, 수요일 참여 | 50,000 × 3 ÷ 5 | 30,000원 |
| 수요일 참여 후 남은 3일 중 공휴일 1일 | 50,000 × 2 ÷ 5 | 20,000원 |

공휴일과 합류 전 날짜는 분자 `R_i`에서 제외하지만 분모 `N`에는 포함한다. 따라서 기준금액은 고정되고 참여자별 적용한도만 달라진다.

### 5.4 중도 합류일

- 진행 중 합류하면 방 시간대 기준 합류한 달력 날짜를 한도 일수에 포함한다.
- 합류일이 공휴일이면 그 날짜는 포함하지 않고 다음 유효일부터 계산한다.
- 합류 확정 시각 전의 지출을 챌린지에 소급 등록할 수 없다.
- 남은 유효일이 하나도 없으면 참여할 수 없다.
- 인원이 추가돼도 기존 참여자의 적용한도는 바뀌지 않는다.

### 5.5 달성

- 개인 달성: `X_i ≤ L_i`
- 개인 초과: `X_i > L_i`
- 전체 완주: 정산 완료 시 유효 참여자 전원이 개인 달성

수입은 입력하거나 적용한도를 늘리는 수단으로 사용하지 않는다.

## 6. 정원, 초대, 참여자

### 6.1 정원

- 방 생성 시 방장 포함 최초 정원을 설정한다.
- 방장은 서비스 최대 정원 안에서 정원을 늘릴 수 있다.
- 현재 활성 참여자 수보다 작게 정원을 줄일 수 없다.
- MVP 서비스 최대 정원 권장값은 활성 멤버 10명이다.
- 마지막 한 자리에 여러 요청이 동시에 들어오면 서버가 먼저 확정한 한 명만 참여에 성공한다.

### 6.2 코드와 초대 링크

- 참여 코드는 혼동 문자를 제외한 대소문자 비구분 6자리 영문·숫자 조합이다.
- 코드와 링크는 대기·진행 상태에서 유효하고 종료 경계 `E`에 만료된다.
- 방장은 `E` 전까지 코드를 재발급할 수 있고 이전 코드는 즉시 무효가 된다.
- 활성 멤버는 유효한 코드나 초대 링크를 다른 사람에게 공유할 수 있다.
- 반복적인 잘못된 코드 입력에는 속도 제한을 둔다.

### 6.3 참여 확인

참여 확정 전 다음 정보를 보여준다.

- 방 이름, 기간, 기준금액
- 전체 선택일과 제외 공휴일
- 현재 멤버 수와 정원
- 합류일, 남은 유효일, 본인 적용한도 계산식
- 기존 피드의 지출·사진·댓글까지 볼 수 있다는 공개 범위

진행 중 합류자에게는 `늦게 합류` 배지를 표시한다. 기존 참여자에게도 새 참여자의 닉네임, 합류일, 적용한도를 알린다.

### 6.4 참여 상태와 나가기

참여 상태는 `활성`, `중도 이탈`, `강제 퇴장`, `계정 탈퇴`로 구분한다.

- 참여자는 진행 중 나갈 수 있으나 같은 챌린지에 다시 참여할 수 없다.
- 기존 지출과 댓글은 대화·감사 맥락을 위해 `나간 참여자`로 남긴다.
- 중도 이탈·강제 퇴장·계정 탈퇴 사용자는 왕관 후보와 전체 완주 분모에서 제외한다.
- 방장이 나가려면 다른 활성 참여자에게 방장 권한을 넘겨야 한다.
- 정상 이전 없이 방장이 탈퇴하면 가장 먼저 참여한 활성 멤버에게 자동 승계한다.

## 7. 지출 기록과 계산

### 7.1 입력 항목

지출 기록은 다음 항목을 가진다.

- 금액
- 발생 일시
- 카테고리
- 메모
- 사진
- 연결 챌린지
- 작성자, 생성·수정 시각, 동기화 상태

수입 유형이나 수입 입력 화면은 제공하지 않는다.

### 7.2 고정 카테고리

MVP 카테고리는 아래 6개만 제공한다.

1. 점심
2. 커피
3. 간식
4. 저녁
5. 필수품
6. 사치품

`기타`와 사용자 정의 카테고리는 제공하지 않는다.

### 7.3 유효 챌린지 지출

아래 조건을 모두 만족할 때 `X_i`에 포함한다.

1. 기록 당시 사용자가 해당 방의 유효 참여자다.
2. 해당 방에 명시적으로 연결했다.
3. 발생 시각이 `[S, E)` 안이며 합류 확정 시각 이상이다.
4. 발생 날짜가 유효 챌린지 날짜 `D`에 포함된다.
5. 금액이 유효한 KRW 정수다.
6. 사진 업로드가 `C` 전에 완료됐다.
7. `C` 시점에 삭제 또는 별도의 계산 제외 상태가 아니다.

같은 원본 지출을 내 기록과 방 피드에 복제하지 않고 하나의 기록을 참조한다.

### 7.4 지출 수정과 삭제

- 진행과 보정 상태에서 작성자만 본인 지출을 수정·삭제할 수 있다.
- 방장도 다른 참여자의 금액을 바꿀 수 없다.
- 금액, 발생 일시, 사진, 카테고리를 바꾸면 `수정됨`과 수정 시각을 표시한다.
- 정확히 `C`가 되면 입력·수정·삭제·사진 재업로드 요청을 차단한다.
- `C` 전에 서버가 수신하지 못한 로컬 변경은 최종 결과에 반영하지 않는다.
- 보정 마감 시 모든 미완료 업로드를 `실패·결과 제외`로 종결한다.
- 정산·완료 상태에서는 운영상 오류 정정 외에 사용자 지출을 바꾸지 않는다.

### 7.5 환불

별도 환불 유형은 첫 MVP에 포함하지 않는다. `C` 전에는 원래 지출을 수정·삭제하고, `C` 이후 환불은 완료된 챌린지 결과를 다시 계산하지 않는다.

## 8. 사진 정책

### 8.1 첨부

- 챌린지 지출은 사진 정확히 1장이 필수다.
- 카메라 촬영과 앨범 선택을 모두 허용한다.
- 저장 전 미리보기와 교체를 제공한다.
- 음식, 구매 물품, 영수증, 결제 화면처럼 지출을 기억할 수 있는 이미지를 허용한다.
- 사진은 지출의 진위를 자동 보증하지 않는다.

### 8.2 업로드와 오프라인

- 서버 저장 전 기록은 `업로드 중` 또는 `동기화 대기`다.
- 대기 기록은 본인 화면의 임시 합계에만 표시하고 공식 합계에는 넣지 않는다.
- 같은 요청 식별자로 재시도해 중복 지출을 만들지 않는다.
- `C`에 업로드가 끝나지 않은 기록은 실패로 종결하고 최종 결과에서 제외한다.

### 8.3 안전과 개인정보

- 업로드 시 위치 등 불필요한 사진 메타데이터를 제거한다.
- 얼굴, 주소, 카드번호 등 민감 정보 노출 가능성을 등록 전에 안내한다.
- 다른 사용자의 화면 캡처를 완전히 막을 수 없다는 점을 공유 전에 알린다.
- 유해 사진을 숨겨도 지출 금액은 유지하며 허위 지출로 확정된 경우에만 별도 계산 제외한다.

## 9. 피드, 댓글, 답글

### 9.1 열람과 작성

- 유효한 방 구성원은 합류 전 기록을 포함한 해당 방 피드 전체를 볼 수 있다.
- 지출 피드는 서버 게시 시각 최신순으로 정렬하고 실제 발생 시각을 별도로 표시한다.
- 댓글은 지출에 종속되며 본문은 공백 제외 1~500자다.
- 댓글은 진행, 보정, 정산 상태에서 작성할 수 있다.
- 완료 전환 시 신규 댓글과 수정은 잠그고 기존 대화는 읽기 전용으로 보관한다.

### 9.2 답글

- 댓글을 길게 누르고 `답글`을 고르면 입력창 위에 작성자와 원문 일부를 읽기 전용 인용 칩으로 표시한다.
- 원문을 사용자의 입력 본문에 직접 복사하지 않는다.
- 답글은 하나의 메시지만 참조하는 1단계 구조다.
- 원문이 삭제되면 `삭제된 메시지에 대한 답글`로 표시한다.

### 9.3 실시간 처리

- 서버 수신 시각과 서버 순서를 최종 기준으로 한다.
- 전송 중, 성공, 실패 상태를 구분한다.
- 요청 식별자로 중복 생성을 막는다.
- 연결 복구 후 누락 메시지를 다시 동기화한다.
- 타이핑 상태, 접속 상태, 메시지별 읽음 확인은 제공하지 않는다.

### 9.4 수정과 삭제

- 본인 댓글만 작성 후 5분 안에 수정할 수 있다.
- 진행·보정·정산에서는 본인 댓글을 소프트 삭제할 수 있다.
- 완료 후에도 개인정보 삭제 요청은 가능하며 본문을 `삭제된 메시지`로 바꾼다.
- 삭제된 댓글의 답글 관계와 운영 신고 기록은 보존한다.

## 10. 왕관 정책

- 후보는 `C` 시점의 활성 멤버다.
- 비교값은 각 참여자의 실제 원화 남은 금액 `M_i`다. `abs(M_i)`가 아니라 부호를 유지한 값을 비교한다.
- `M_i`가 가장 큰 참여자의 닉네임 앞에 `👑`을 표시한다.
- 동률이면 해당 사용자 모두에게 공동 왕관을 표시한다.
- 진행·보정 중 지출·수정·삭제·합류·이탈이 확정될 때마다 실시간 재계산한다.
- 정산에서는 `잠정 왕관`으로 고정하고 `F`에 최종 왕관으로 확정한다.
- 적용한도가 다른 중도 합류자도 사용자 결정에 따라 같은 실제 원화 남은 금액 기준으로 비교한다.
- 전원이 한도를 초과해도 남은 금액이 가장 큰 활성 멤버에게 왕관을 표시한다.
- 왕관은 달성 인증이나 금전적 보상이 아니다.

## 11. 완료 기록 보관

### 11.1 사용자 경험

- `F`가 되면 방을 삭제하거나 복제하지 않고 `ARCHIVED` 상태로 바꾼다.
- 홈의 `지난 챌린지`에서 연도·월별로 묶어 보여준다.
- 챌린지명 검색과 참여 여부·달성 여부 필터를 제공한다.
- 상세 화면은 고정 조건, 공휴일, 참여자별 계산 근거, 지출, 사진, 댓글, 결과를 읽기 전용으로 보여준다.
- 사용자는 자신의 목록에서 숨기거나 다시 표시할 수 있다.

### 11.2 데이터 보관 구조

완료 시 다음 스냅샷을 만든다.

- 방 계산 조건: 기간, 기준금액, 선택일, 공휴일 버전과 제외일
- 멤버 결과: 합류일, 적용한도, 지출 합계, 남은 금액, 달성, 왕관
- 완료 상태: `C` 잠금 시각, `F` 확정 시각

지출과 댓글 원본은 기존 식별자를 유지하고 수정 불가 상태로 전환한다. 사진은 객체 저장소에 최적화 원본과 썸네일을 보관하며 목록에서는 썸네일, 상세에서는 원본을 지연 로딩한다. 긴 목록은 페이지 단위로 불러온다.

### 11.3 보유 기간과 예외

- 완료 방에는 자동 만료일을 두지 않고 서비스 운영 중 계속 보관한다.
- 계정 탈퇴, 이용자 삭제 요청, 법적 의무, 유해 콘텐츠 조치는 예외다.
- 공유 결과의 금액·합계는 비식별 스냅샷으로 유지할 수 있다.
- 사진·메모·댓글 본문 삭제 요청은 실제 내용을 제거하고 자리표시자만 남긴다.
- 정확한 백업 삭제 주기와 법적 보존 기간은 출시 전 개인정보 처리방침에서 확정한다.

## 12. 알림

- 챌린지 시작 10분 전과 시작
- 새 참여자 합류와 정원 마감
- 내 지출의 새 댓글과 내 댓글에 대한 답글
- 개인 적용한도 50%, 80%, 100% 도달
- 업로드·동기화 실패
- `E`: 보정 시작
- `C-2h`: 지출 보정 마감 2시간 전
- `C`: 지출 변경 잠금과 정산 시작
- `F`: 결과·왕관 확정과 지난 챌린지 이동

왕관이 바뀔 때마다 푸시하지 않는다. 사용자는 전체 알림과 방별 알림을 끌 수 있다.

## 13. 신고, 차단, 공개 범위

### 13.1 공개 범위

| 데이터 | 본인 | 같은 방 멤버 | 외부/다른 방 |
|---|---:|---:|---:|
| 방에 연결하지 않은 개인 지출 | 열람 | 불가 | 불가 |
| 방에 연결한 지출 금액·사진·메모 | 열람 | 열람 | 불가 |
| 적용한도·사용·남은 금액·왕관 | 열람 | 열람 | 불가 |
| 댓글·답글 | 열람 | 열람 | 불가 |
| 로그인·기기 정보 | 제한적 열람 | 불가 | 불가 |

### 13.2 신고와 차단

- 지출 사진, 메모, 댓글, 프로필을 신고할 수 있다.
- 신고 사유는 괴롭힘, 혐오·욕설, 개인정보 노출, 부적절한 이미지, 스팸, 기타다.
- 같은 방에서 차단해도 계산에 필요한 금액과 참여 상태는 보이되 사진·댓글은 흐림 처리할 수 있다.
- 신고자 정보는 피신고자와 일반 멤버에게 공개하지 않는다.
- 콘텐츠 노출 상태와 금융 계산 상태를 분리한다.

## 14. 오류, 오프라인, 동시성

| 정산 중 서버 장애 | 모든 참여자에게 같은 시간만큼 `F`를 연장하거나 판정 보류 |

## 15. 운영자 도구 최소 요구사항

- 방, 참여자, 적용한도 계산 근거 조회
- 공휴일 스냅샷과 데이터 버전 조회
- 지출·사진·댓글·답글 식별자 조회
- 신고 큐와 콘텐츠 임시 숨김
- 계정·방 접근 제한
- 수정·삭제·제재 감사 로그
- `E/C/F` 상태 전환과 실패 작업 모니터링
- 사진 업로드, 실시간 메시지, 알림 실패 모니터링

운영자는 개인 전용 지출을 일반적으로 열람하지 못하며 필요한 경우에만 사유와 로그를 남겨 제한적으로 접근한다.

## 17. 정책 수용 기준

- `S/E/C/F` 경계에서 권한이 정확히 바뀐다.
- 월~금 챌린지는 토요일 12:00에 지출이 잠기고 월요일 00:00에 완료된다.
- 생성된 방의 기간과 기준금액은 방장도 수정할 수 없다.
- 공휴일 1일이 있는 5일·50,000원 방의 시작 전 멤버 적용한도는 40,000원이다.
- 공휴일이 없는 같은 방의 수요일 합류자 적용한도는 30,000원이다.
- 수요일 이후 공휴일 1일이 있으면 해당 합류자의 적용한도는 20,000원이다.
- 중도 합류자는 합류 확정 시각 이전 지출을 소급 등록할 수 없다.
- 카테고리는 6개 고정값만 저장된다.
- 수입 입력 API와 화면이 존재하지 않는다.
- 남은 금액 최대 멤버에게 왕관이 표시되고 동률은 공동 표시된다.
- `C` 이후 사용자 지출 변경이 결과를 바꾸지 않는다.
- `F` 이후 방·계산 근거·사진·댓글·결과를 지난 챌린지에서 읽을 수 있다.
- 방 구성원이 아닌 계정은 코드 추측만으로 기록을 열람할 수 없다.
- 오프라인 재전송 후에도 지출과 댓글이 중복 생성되지 않는다.
